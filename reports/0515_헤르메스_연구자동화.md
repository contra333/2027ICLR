## 1. 현재 연구에서 느끼는 문제점

현재 ICLR 2027 연구는 NeurIPS 2026에서 관찰한 현상을 더 넓은 optimizer family와 feature geometry 관점으로 확장하는 단계에 있습니다. 핵심 질문은 optimizer choice와 weight-decay coupling이 penultimate feature geometry 및 Neural Collapse regime을 어떻게 바꾸고, 이 변화가 Mahalanobis, DDU/GMM, kNN, ViM, NC/prototype detector에 어떻게 다르게 읽히는가입니다.

이 단계에서 가장 큰 병목은 GPU 실험을 바로 많이 돌리는 것이 아니라, 실험 정의와 점검 체계를 먼저 믿을 수 있게 만드는 데 있습니다. optimizer 비교가 공정한지, metric 이름과 score direction이 프로젝트 계약과 선행연구 관행에 맞는지, feature-based OOD detector가 같은 feature cache와 같은 기준으로 비교되는지를 확인하지 않으면, 이후 200 epoch 실험 결과가 나와도 논문에서 방어하기 어렵습니다.

| 현재 문제 | 설명 | 먼저 해결해야 하는 이유 |
|---|---|---|
| 실험 조건의 공정성 | dataset, architecture, augmentation, epoch, scheduler, seed, weight decay policy가 optimizer 외 조건으로 흔들릴 수 있습니다. | optimizer-induced geometry라는 주장에서는 optimizer 외의 차이가 confound가 됩니다. |
| metric 정의와 방향 | OOD score에서 higher가 ID-like인지 OOD-like인지, `fpr95`, `aupr_in`, ECE 계산이 일관되는지 확인해야 합니다. | score direction이 뒤집히면 AUROC/FPR95 해석이 완전히 달라질 수 있습니다. |
| detector 구현 범위 | Mahalanobis, DDU/GMM, kNN, ViM, NC/prototype detector가 같은 기준으로 비교되는지 점검해야 합니다. | reviewer가 detector family coverage와 tuning protocol을 문제 삼을 수 있습니다. |
| 서버 운영의 단절 | 긴 점검과 반복 audit은 연구자가 컴퓨터 앞에 있을 때만 진행되면 속도가 느려집니다. | 밤 시간에도 문서 읽기, 코드 대조, 보고서 초안 작성이 이어져야 다음 날 바로 판단할 수 있습니다. |

## 2. 이를 해결하기 위한 Hermes Agent 도입

이 문제를 줄이기 위해 Hermes Agent를 연구 운영 보조 장치로 도입하였습니다. Hermes Agent는 논문 결론을 대신 내리는 도구가 아니라, 연구자가 자리에 없는 시간에도 연구 문서와 코드를 읽고, 점검 보고서와 실행 명령 초안을 남기는 보조 프로그램입니다.

도입 목적은 단순히 챗봇을 하나 더 사용하는 것이 아닙니다. 연구 context가 Slack channel, 사설 서버 workspace, repo clone, daily briefing output에 누적되도록 만들고, 반복적인 문헌 조사, 실험 계획 점검, metric audit, 로그 분석을 밤 시간에도 진행할 수 있게 하는 것입니다.

| 구분 | Hermes Agent가 맡는 역할 | 연구자가 맡는 역할 |
|---|---|---|
| 밤 시간 연구 운영 | Slack 지시를 받아 repo와 문서를 읽고 audit report 초안을 남깁니다. | 연구 방향, 우선순위, 최종 판단을 결정합니다. |
| 실험 전 점검 | code/config/metric/source boundary를 정적으로 대조합니다. | 실제 코드 수정, 테스트, commit, GPU 실행을 승인합니다. |
| 반복 작업 | 문헌 후보 정리, server command 초안, manifest check list를 만듭니다. | 논문 주장으로 사용할 수 있는지 evidence level을 판단합니다. |
| 보고 준비 | daily research log와 audit 결과를 보고용 구조로 정리합니다. | 보고서의 해석 강도와 표현을 최종 조정합니다. |

## 3. Hermes Agent가 무엇인지

서버나 운영체제 지식이 많지 않은 독자도 Hermes Agent를 다음과 같이 이해할 수 있습니다.

Hermes Agent는 항상 켜져 있는 원격 컴퓨터 위에서 돌아가는 연구 보조 프로그램입니다. 연구자가 Slack에 지시를 남기면, Hermes가 연구 repo와 문서를 읽고, 필요한 점검을 수행한 뒤, 보고서 초안이나 실행 명령 초안을 남깁니다. 여기서 Slack은 연구 지시를 남기는 대화방이고, repo는 연구 코드와 문서를 Git으로 관리하는 작업 폴더입니다.

| 용어 | 짧은 설명 | 이 연구에서의 역할 |
|---|---|---|
| Slack | 여러 사람이 대화방처럼 메시지를 남기는 협업 도구입니다. | `#2027iclr` channel에 연구 지시와 결과 기록을 남깁니다. |
| VPS | 개인이 빌려 쓰는, 항상 켜져 있는 원격 컴퓨터입니다. | 연구자의 노트북이 꺼져 있어도 Hermes가 계속 대기할 수 있게 합니다. |
| repo | 코드, 설정, 문서를 Git으로 관리하는 연구 작업 폴더입니다. | Hermes가 ICLR2027 연구 파일을 읽고 audit할 기준 자료입니다. |
| Docker sandbox | 프로그램을 본체와 분리된 실행 공간에서 돌리는 방식입니다. | Hermes가 분석 작업을 하더라도 서버 본체와 연구 파일을 더 안전하게 보호합니다. |
| GPU 서버 | 딥러닝 학습처럼 계산량이 큰 일을 돌리는 별도 서버입니다. | Hermes가 직접 대규모 학습을 마음대로 실행하지 않고, 실행 계획과 명령 초안을 준비합니다. |

전체 흐름은 단순합니다. 연구자가 Slack에 "이 repo를 읽고 metric 정의가 맞는지 점검해라"라고 지시하면, Hermes가 사설 서버 안의 분리된 실행 공간에서 repo를 읽고, 점검 결과를 Markdown 보고서 형태로 남깁니다. 이후 로컬 Codex로 실제 수정과 테스트를 수행하거나, GPU 서버에서 실행할 명령을 확정합니다.

## 4. 왜 이런 Agent 방식이 요즘 많이 사용되는가

요즘 agent 방식이 많이 사용되는 이유는 AI가 대화를 잘하기 때문만은 아닙니다. 연구와 개발에서는 파일을 읽고, 코드와 문서를 대조하고, 명령을 제안하고, 결과를 보고서로 정리하는 반복 작업이 많습니다. Agent는 이런 작업을 하나의 흐름으로 이어 줄 수 있습니다.

일반 챗봇은 질문에 답하는 데 집중합니다. 반면 Hermes 같은 agent는 repo 안의 문서와 코드를 읽고, 긴 점검 절차를 수행하고, 다음에 연구자가 실행할 명령과 보고서 초안을 남기는 데 초점이 있습니다. 즉 "대답하는 AI"보다 "연구 운영을 계속 진행시키는 보조 작업자"에 가깝습니다.

| 장점 | 연구에서의 의미 | 주의점 |
|---|---|---|
| 비동기 작업 | 밤에도 문헌 후보 조사, config 점검, metric audit이 진행될 수 있습니다. | 최종 판단과 논문 claim은 연구자가 해야 합니다. |
| 파일 기반 점검 | repo, source note, metric contract, server workflow 문서를 함께 읽을 수 있습니다. | 읽은 파일과 읽지 않은 파일을 보고서에 구분해야 합니다. |
| 반복 audit 자동화 | 같은 형식으로 누락, 위험, 다음 명령을 정리할 수 있습니다. | 자동 실행 권한을 과하게 주면 위험하므로 금지 규칙이 필요합니다. |
| context 축적 | Slack과 workspace에 연구 운영 기록이 남습니다. | 중요한 결과는 repo 문서와 manifest로 다시 정리해야 합니다. |

## 6. 오늘 연구에 접목한 방식

Hermes를 처음 연결한 연구 작업은 ICLR2027 pre-flight audit입니다. 여기서 pre-flight audit은 GPU 시간을 쓰기 전에 실험 설정과 metric 정의가 논문에서 방어 가능한 상태인지 확인하는 절차입니다.

| audit 항목 | Hermes에게 맡긴 확인 | 연구에서의 의미 | 다음 행동 |
|---|---|---|---|
| 학습설정 공정성 audit | dataset, architecture, augmentation, epoch, scheduler, seed, total weight decay, feature layer가 고정되는지 확인합니다. | optimizer 비교에서 confound를 줄입니다. | 200 epoch config 확정 전 반영합니다. |
| optimizer semantics audit | `code/IMPLEMENTATION_CONTRACT.md`와 optimizer factory/code를 대조합니다. | Adam, AdamW, coupled-decoupled WD 축의 의미가 흐려지지 않게 합니다. | endpoint semantics 문서와 코드 일치를 확인합니다. |
| metric definition audit | `higher = ID-like`, `ID = 1`, `OOD = 0`, `fpr95`, `aupr_in`, ECE, Energy sign을 확인합니다. | OOD metric 해석이 뒤집히는 위험을 막습니다. | evaluator output contract를 점검합니다. |
| OOD detector gap matrix | logit, feature, density, hybrid, NC/prototype detector별 구현/누락/위험을 분류합니다. | reviewer objection에 대비해 detector family coverage를 확인합니다. | Tier 0 evaluator를 보강합니다. |
| Geometry / Neural Collapse metric audit | feature geometry와 NC component naming이 report contract와 맞는지 확인합니다. | optimizer-induced geometry claim의 측정 축을 안정화합니다. | metric naming과 output schema를 점검합니다. |
| source boundary 확인 | optimizer/NC paper, NECO, Mahalanobis++, ViM, smoke validation의 evidence level을 분리합니다. | 문헌 근거, smoke validation, ICLR hypothesis를 한 증거층처럼 사용하지 않게 합니다. | report와 paper text에서 경계를 유지합니다. |

Hermes에게 요구한 보고서 형식은 `Executive Summary`, `Files Read`, `Training Setup Fairness Audit`, `Optimizer Semantics Audit`, `Metric Definition Audit`, `OOD Detector Implementation Gap Matrix`, `Geometry / Neural Collapse Metric Audit`, `High-Risk Ambiguities`, `Recommended Fix Order`, `Commands I Should Run Locally or on GPU Server Later`로 구성하였습니다.

## 7. 현재 증거 수준과 주의점

현재 증거는 여러 층으로 분리해야 합니다. 이 분리는 보고서와 논문 작성 모두에서 중요합니다.

| 증거층 | 현재 상태 | 사용할 수 있는 주장 | 주의점 |
|---|---|---|---|
| NeurIPS 2026 prior | 확정 | SAM에서 output reliability와 feature-based uncertainty가 분리될 수 있습니다. | ICLR 확장 결과와 섞지 않습니다. |
| 외부 문헌 | 확정/해석 | optimizer choice가 NC/geometry에 영향을 주고, OOD detector family와 protocol 설계가 중요합니다. | downstream detector behavior는 본 연구의 실험으로 검증해야 합니다. |
| M1 smoke validation | 확정 | train, cache, eval, manifest pipeline이 최소 단위로 동작합니다. | 2 epoch smoke metric은 paper-level evidence가 아닙니다. |
| Hermes audit | 해석/운영 | full-scale 실험 전에 code/config/metric/source boundary 위험을 찾는 운영 점검입니다. | audit 결과 자체가 논문 claim의 실험 근거는 아닙니다. |
| 자동 실행 권한 | 위험 | Hermes가 편리하더라도 자동 push, 자동 GPU 실행, 대량 삭제 권한은 제한해야 합니다. | 편의성보다 연구 파일과 서버 안전을 우선합니다. |
| full-scale 결과 | 누락 | 아직 200 epoch, multi-seed, matched-protocol 결과는 확보되지 않았습니다. | ICLR main claim은 이 결과가 나온 뒤에만 강하게 쓸 수 있습니다. |
| ICLR main claim | 가설 | optimizer-induced geometry regime이 detector family별 성공/실패를 설명할 수 있습니다. | 200 epoch, multi-seed, matched-protocol 결과가 필요합니다. |

따라서 Hermes 도입의 의미는 "새로운 연구 결과가 이미 나왔다"가 아닙니다. 정확한 의미는 "논문급 실험을 시작하기 전에, 실험 정의와 구현 위험을 더 체계적으로 점검할 수 있는 운영 구조를 구축하였다"입니다.

## 9. 오늘 추가로 정한 보고 workflow
