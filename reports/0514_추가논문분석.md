## 0513 Survey Intent

`0513_metric_OOD_survey.md` shows that the user was trying to answer four connected questions.

1. What are the current representative OOD detector families?
2. Which metrics and detector scores should enter the ICLR paper, beyond AUROC and the existing Mahalanobis/DDU focus?
3. If optimizer-induced geometry hurts feature-based detectors, can normalization, hybrid scores, NC-based scores, or other newer methods simply fix the issue?
4. How should the paper position itself in the recent OOD literature without becoming an all-detectors benchmark paper?

The useful conclusion from the 0513 note is not "add every OOD detector." It is that the experiment design should cover enough detector families to show which parts of representation geometry matter:

- logit-only reliability: `msp`, `maxlogit`, `energy_id_score`, `neg_entropy`
- normalized feature controls: `mahalanobis_l2`, `knn_l2`
- local geometry: `knn`
- covariance and density assumptions: `gmm_ddu_*`
- hybrid logit-feature response: `vim_id_score`
- NC/prototype geometry response: `nc_prototype_cosine`, `nc_residual_subspace`, `nc_classifier_alignment`, eventual `neco_lite`

## Project Interpretation

The added literature strengthens the paper if we frame the contribution as a detector-family response analysis, not a new OOD detector benchmark.

Recommended paper message:

> Optimizers do not merely change accuracy or calibration. They induce different feature geometry regimes, and output-based, hybrid, distance-based, density-based, and NC-based OOD scores respond differently to those regimes.

This lets us answer the likely reviewer objection:

> If another detector fixes the degradation, that is not a contradiction. The recovery pattern tells us which geometry component was damaged or preserved.

Examples:

| Result pattern | Interpretation |
|---|---|
| `mahalanobis` degrades but `mahalanobis_l2` recovers | Feature norm or covariance scale is a major channel. |
| `mahalanobis`, `mahalanobis_l2`, and `knn_l2` all degrade | Angular/local class geometry is affected, not just feature norm. |
| `gmm_ddu_full` degrades but `gmm_ddu_tied/diag/shrinkage` is stable | Covariance estimation or class covariance conditioning is the main failure mode. |
| `vim_id_score` stays close to Energy | Logit/energy term compensates for residual feature geometry damage. |
| `vim_id_score` follows Mahalanobis/kNN | Feature residual or principal subspace geometry is part of the optimizer-sensitive channel. |
| NC/prototype scores shift while accuracy is stable | The optimizer changes detector-relevant NC geometry without necessarily hurting classification. |

| Category | Add or keep | Why |

| Distance detectors | `mahalanobis`, `mahalanobis_l2`, `knn`, `knn_l2` | Separates Gaussian/covariance assumptions, feature norm effects, and local neighborhood geometry. |
| Density controls | `gmm_ddu_tied`, `gmm_ddu_diag`, `gmm_ddu_shrinkage`, diagnostic `gmm_ddu_full`, `gmm_ddu_pca` | Tests whether DDU failure is raw geometry or covariance estimation instability. |
| Hybrid detector | `vim_id_score` | Tests whether feature-residual plus logit scores behave like logit-only or feature-only methods. |
| NC/prototype diagnostics | `ncc_distance`, `nc_prototype_cosine`, `nc_residual_subspace`, `nc_classifier_alignment` | Connects detector behavior to the NC story without overclaiming official NECO. |

### Priority 1: appendix robustness

Use these only after the main detector-family story is stable:

- `react`, `ash`, `scale`, `rankfeat`, `dice`, `she`, `gen`
- They are useful for asking whether activation shaping or feature clipping repairs the degradation.
- They should not replace the main raw-representation evidence because they intervene on the detector-side representation.

### Priority 2: defer from the first evidence package

- ODIN and GradNorm as main evidence
- OpenMax as main evidence
- OE, VOS, NPOS, CIDER, DRL, or methods requiring auxiliary outliers or altered training objectives
- CLIP/DINOv2 prompt-based OOD as proof of the from-scratch optimizer-geometry claim

## Experimental Settings to Add or Lock

### Main controlled setting

- ID datasets: CIFAR-10 first, then CIFAR-100.
- Architectures: standard ResNet-18 and WRN-28-10 first.
- Optimizers: SGD, SGDW, Adam, AdamW, controlled `adam_coupled_decoupled`, SAM, ASAM, GSAM.
- Checkpoints: final checkpoint for geometry and detector analysis; best-validation checkpoint only as appendix.
- Feature cache: one shared cache per checkpoint for all post-hoc detectors.

### OOD split structure

| Split type | Candidate datasets | Purpose |
|---|---|---|
| Far-OOD | SVHN, Texture, MNIST/FashionMNIST where appropriate | Confirms behavior under easier semantic shifts. |
| Near-OOD | CIFAR-100 for CIFAR-10 ID, TinyImageNet, NINCO later | Tests the harder regime where geometry-sensitive detectors matter most. |
| Covariate shift / full-spectrum | CIFAR-10-C or CIFAR-100-C later | Keeps OpenOOD-style full-spectrum framing separate from pure semantic OOD. |

### Detector tuning policy

- Main evidence must not tune detector hyperparameters using OOD test AUROC, AUPR, or FPR95.
- ID train fits detector parameters such as means, covariance, PCA basis, kNN reference set, NC prototypes.
- ID validation may select explicitly declared parameters such as temperature, shrinkage alpha, or PCA dimension.
- If OOD tuning is explored, it belongs in a tuning-sensitivity appendix and must be labeled as such.

## Related-Work Structure

Recommended structure for the paper:

1. OOD detector landscape
   - Output/logit, distance, density, hybrid/feature-shaping, NC/prototype, and large-pretrained-model methods.
2. Benchmark and protocol caveats
   - OpenOOD v1.5, Task-Oriented Survey, and OOD/OSR critical analysis.
   - Emphasize near/far OOD, semantic/covariate shift, full-spectrum settings, and tuning-set sensitivity.
3. Representation and training interaction
   - Systematic representation/training-shift analysis and One Model, Many Behaviors.
   - Position our work as controlled optimizer-geometry analysis rather than a detector proposal.
4. Neural Collapse and optimizer geometry
   - Optimizer choice / NC paper supports optimizer -> geometry.
   - NECO supports NC geometry as detector-relevant.
   - Our experiments must supply the downstream optimizer -> detector evidence.
5. Pretrained and VLM regimes
   - Acknowledge CLIP/VLM as current direction.
   - Keep it as extension or regime analysis, not the main proof.

## Local Hypotheses to Test

These are not confirmed results yet.

1. SAM or decoupled-WD adaptive optimizers may preserve logit reliability while changing feature geometry enough to hurt feature-density detectors.
2. If `mahalanobis_l2` and `knn_l2` recover strongly, feature norm variation is a key causal channel.
3. If normalized feature methods still degrade, angular prototype geometry, class separation, local density, or NC alignment is more important.
4. If `vim_id_score` remains stable, hybrid scores can mask feature degradation through logit terms.
5. If `vim_id_score` degrades with Mahalanobis/kNN, optimizer effects reach principal subspace or residual geometry.
6. If `gmm_ddu_shrinkage` improves over `gmm_ddu_full`, covariance estimation is a separate confound that must be controlled.
7. If NC/prototype diagnostics move with detector degradation, the paper can argue that "Neural Collapse is not one geometry" in detector-relevant terms.

## Risks and Source Boundaries

- Do not cite the optimizer/NC paper as direct evidence for Energy, Mahalanobis, kNN, DDU, or ViM behavior.
- Do not call `neco_lite` official NECO unless the exact paper or official-code formula is implemented as `neco_full`.
- Do not claim smoke-run metrics as ICLR evidence. Current M1A/M1B metrics are pipeline validation only.
- Do not let detector differences come from different feature extraction paths. Use shared caches.
- Do not let OOD validation choices determine main detector rankings.
- Do not make VLM/CLIP claims central until a separate pretrained-regime experiment exists.
