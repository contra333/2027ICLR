연구 방향

이 프로젝트의 목표는 “최신 OOD detector를 많이 돌렸다”가 아니다. 핵심은 optimizer가 만든 representation geometry regime을 detector family가 서로 다르게 읽는다는 점을 통제된 실험으로 보여주는 것이다.

현재 thesis: optimizer improvement는 reliability에 대해 하나의 의미만 갖지 않는다. 어떤 optimizer는 logit confidence나 calibration에는 좋아 보이지만, feature covariance, class mean, neighborhood, NC component를 읽는 detector에는 불리한 geometry를 만들 수 있다.

근거 축 쓸 수 있는 주장 주의할 점 먼저 볼 파일
NeurIPS 2026 prior SAM이 accuracy/calibration/logit reliability를 유지하거나 개선하면서도 Mahalanobis/DDU를 악화시킬 수 있다는 prior 관찰. 새 ICLR claim으로 일반화하려면 새 controlled experiment가 필요하다. 소스/neurips2026_paper_context.md
Optimizer / NC paper optimizer choice와 coupled/decoupled weight decay가 Neural Collapse / representation geometry에 영향을 준다는 축. 이 논문만으로 Energy, Mahalanobis, kNN, DDU 결과를 직접 주장하면 안 된다. 소스/GPT_PROJECT_SOURCE_GUIDE.md
OpenOOD v1.5 OOD benchmark, metric convention, near/far OOD 평가 맥락. 우리 논문은 OpenOOD 전체 재현 논문이 아니다. OpenOOD AI_README
NECO Neural Collapse geometry가 post-hoc OOD detector로 쓰일 수 있다는 근거. optimizer가 NECO 성능을 바꾼다는 주장의 직접 근거는 아니다. NECO AI_README
Mahalanobis++ L2 feature normalization이 Mahalanobis-style OOD에서 feature norm/covariance confound를 줄이는 control이 될 수 있음. mahalanobis_l2는 full reproduction이 아니라 Mahalanobis++-motivated control로 기록한다. Mahalanobis++ AI_README
ViM feature residual과 logit을 결합한 hybrid OOD scoring 근거. native OOD score와 project ID-like score 방향을 분리해야 한다. ViM AI_README

기본 작업 흐름

1. 로컬 계획 AI_CONTEXT.md, 소스/INDEX.md, 관련 AGENTS.md를 읽고 목표를 좁힌다.
2. 코드/config 준비 code/와 configs/에 서버 실행 가능한 최소 단위를 만든다.
3. Git으로 서버 전달 로컬 main 또는 exp/<short-name> branch를 push하고 서버에서 pull한다.
4. 서버 실행 GPU 서버에서 run ID, config hash, commit, command, log를 함께 남긴다.
5. 로컬 분석 metrics/logs를 results/raw/로 복사하고 manifest를 만든 뒤 reports/에서 해석한다.

서버 run ID 규칙

서버 실행은 YYYYMMDD_HHMM_<server>_<dataset>_<model>_<optimizer>_seed<seed> 형식의 run_id를 가진다. 예: 20260512_2130_101_cifar10_resnet18_adamw_seed0.

Git으로 관리하는 것과 하지 않는 것

Git에 넣는 것: 실험 코드, config, 문서, source index/manifest, run manifest, 작은 processed summary, figure script, 작은 최종 figure.
Git에서 제외하는 것: raw server run folder, checkpoint, feature dump, large array, temporary log, dataset, local virtual environment.

평가 metric 계약

학습된 checkpoint 하나를 평가할 때는 모든 post-hoc detector가 같은 cached logits/features에서 동작해야 한다. OOD detector score는 프로젝트 전체에서 higher = ID-like, ID label = 1, OOD label = 0으로 통일한다.

범주 주요 output name 해석 규칙
Classification accuracy, nll ID test 기준. accuracy는 높을수록, nll은 낮을수록 좋다.
Calibration ece_15bin, temperature_scaled_ece_15bin legacy TCE 대신 명확한 이름을 사용한다. temperature는 ID validation으로만 fit한다.
OOD aggregate auroc, fpr95, aupr_in aggregate는 보고용이다. OOD test score로 detector hyperparameter를 고르지 않는다.
Logit detectors msp, maxlogit, energy_id_score, neg_entropy energy_id_score는 project convention에 맞춘 ID-like 방향이다.
Feature detectors mahalanobis, mahalanobis_l2, knn, knn_l2, gmm_ddu_* mahalanobis_l2는 Mahalanobis++-motivated control이며, GMM/DDU variants는 분리해서 저장한다.
NC / hybrid ncc_distance, ncc_accuracy, nc4_agreement, vim_id_score ncc_accuracy와 strict nc4_agreement를 섞지 않는다. vim_native_ood_score는 native diagnostic이다.
Geometry nc0_width_norm, nc1, nc2_*_etf, nc3_self_duality, inter_dist_l2, inter_dist_sq legacy nc0, nc3, nc4, inter_dist 이름 대신 revised names를 사용한다.

자세한 정의는 reports/METRIC_DEFINITIONS.md가 source of truth다.
